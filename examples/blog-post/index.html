How PresetCSS turns a flat TOML file into optimized utility CSS

A walkthrough of the parse → resolve → generate → emit pipeline — and why choosing TOML over JavaScript was the right call.

M

Why TOML?

Most CSS-in-JS solutions and utility frameworks ask you to configure them in JavaScript. That means a build step, a Node runtime, and a config file that can run arbitrary code. PresetCSS chose TOML for a simpler reason: what you write is exactly what gets generated.

A flat key-value file maps directly to a flat list of CSS rules. No deep merges, no extend vs override ambiguity, no runtime surprises.

The four stages

1
Parse — TOML → RawPreset struct via serde
2
Resolve — merge defaults, validate tokens, detect collisions
3
Generate — Theme → Vec<Rule>, static utilities, variants
4
Emit — serialize to CSS, optionally minify, tree-shake

Tree-shaking with content scanning

Set content = ["src/**/*.html"] and PresetCSS will scan your templates for used class names using the Aho-Corasick algorithm — O(n) in the size of the content, not the number of tokens. Only classes that appear in your templates are written to the output file.

Near-miss detection means if you write bg-primry instead of bg-primary, you get a warning — not a silent miss.

Variant opt-in

Unlike Tailwind (which generates all variants by default and relies on purge), PresetCSS generates zero variant rules unless you opt in via [variants]. This keeps the output tiny even without content scanning.
